Introduction

LEAKFINDER is a Java-based automated web vulnerability scanner that detects OWASP Top 10 vulnerabilities, information disclosure, and common misconfiguration patterns in web applications. It's designed to be accurate, fast, and require minimal false positives.

Unlike browser-based tools or heavy frameworks, LEAKFINDER operates as a standalone scanner that can be integrated into:

  • CI/CD pipelines (automated testing)
  • Security assessments (manual penetration testing)
  • Bug bounty workflows (recon automation)
  • Scheduled audits (daily/weekly scans)
Design Philosophy: LEAKFINDER prioritizes precision over coverage. It's better to find 5 real vulnerabilities than report 50 false positives.

The Web Security Gap

Web applications are a primary attack surface. According to OWASP and security research:

  • 85% of web apps have OWASP Top 10 vulnerabilities
  • SQL Injection and XSS remain in top 3 after 20 years
  • Information disclosure is endemic (exposed configs, secrets, paths)
  • Manual testing is slow (100+ endpoints → days of work)

Existing tools either generate noise, miss subtle issues, require extensive setup, or don't integrate with CI/CD. LEAKFINDER fills this gap with a low-noise, high-precision scanner that integrates seamlessly into development and security workflows.

Scanner Architecture

LEAKFINDER is modular and layered:

Layer 1: Reconnaissance

Gather information about the target: URL enumeration, pattern extraction, technology detection, and response analysis.

Layer 2: Vulnerability Detection

Run targeted checks for specific vulnerability classes: injection attacks, XSS, CSRF, authentication issues, information disclosure, and business logic flaws.

Layer 3: Validation & Reporting

Verify findings with multiple techniques, assign CVSS-based severity scores, provide remediation guidance, and output in JSON, HTML, CLI, or SARIF formats.

Detection Modules

1. SQL Injection Detector

Tests time-based blind SQLi, boolean-based blind SQLi, error-based SQLi, and union-based SQLi patterns.

Example: Parameter id=1 AND 1=1 returns 100 results; id=1 AND 1=2 returns 0 results → SQL injection confirmed.

2. Cross-Site Scripting (XSS) Detector

Tests multiple XSS vectors: basic payloads, attribute-based, event handlers, and HTML entity bypass.

3. Information Disclosure Detector

Searches for exposed sensitive data: backup files, configuration files, version disclosures, source code exposure, and debug info.

4. Authentication & Authorization Tester

Tests broken session handling, authorization bypasses, authentication bypass, and CSRF token validation.

5. API Security Checker

Specific checks for RESTful/GraphQL APIs: missing authentication, overly permissive CORS, missing rate limiting, and sensitive data in responses.

Scanning Workflow

Phase 1: Crawling (2-5 minutes)

Spider the web application to discover all endpoints.

Phase 2: Technology Detection

Fingerprint the application stack: server, framework, frontend libraries, and database.

Phase 3: Vulnerability Testing (5-15 minutes)

Run detection modules against each endpoint with targeted checks.

Phase 4: Validation & Reporting

Confirm findings and generate output with severity scoring and remediation guidance.

Detection Examples

Example 1: SQL Injection

Vulnerable endpoint: GET /products?id=1

GET /products?id=1' AND SLEEP(5)--+ HTTP/1.1
Host: example.com

If response takes 5+ seconds, SQL injection is confirmed. LEAKFINDER extracts database version, table names, and record counts.

Example 2: XSS in User Input

Vulnerable endpoint: POST /comments

If the XSS payload appears unescaped in the response or is stored in the database, XSS is confirmed.

Example 3: Information Disclosure

LEAKFINDER finds common exposed files: .env, .git/config, backup.sql, config.json. If any return 200 OK, the file exists and contents may leak credentials, API keys, database info.

Usage & Integration

Quick Start

git clone https://github.com/aizen-ackerman/leakfinder
cd leakfinder
mvn clean compile
java -jar target/leakfinder.jar --target https://example.com --output report.html
# Open report.html in browser

CLI Options

--target              Target URL (required)
--scope               Crawl scope (same-domain, subdomain, all)
--modules             Specific modules to run (sql, xss, csrf, auth)
--output              Output file (HTML, JSON, CSV)
--intensity           Scan intensity (light, normal, aggressive)
--concurrency         Number of parallel requests (default: 5)
--depth               Crawl depth limit (default: 3)
--timeout             Request timeout in seconds (default: 10)

CI/CD Integration

Add to GitHub Actions for automated security scanning on every push or pull request.

Deployment Options

  • Standalone JAR: Run locally or in Docker
  • Maven integration: mvn leakfinder:scan
  • Docker: Containerized for consistent environments
  • REST API: HTTP endpoint for remote scanning
Legal Note: Only scan systems you own or have explicit permission to test. Unauthorized scanning violates CFAA and other laws.
