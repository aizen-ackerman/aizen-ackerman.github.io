Introduction

Post-quantum cryptography (PQC) is becoming critical infrastructure as quantum computing advances. However, the transition from classical to quantum-resistant algorithms creates a window of vulnerability: downgrade attacks.

This writeup documents a working proof-of-concept (POC) that demonstrates how an attacker positioned at the network layer can strip Kyber key encapsulation from TLS ClientHello messages, forcing a downgrade to classical (X25519) key exchange. The attack is built using OpenSSL and liboqs, targeting real TLS handshakes.

Research Context: This POC is educational and demonstrates a real threat to hybrid PQC/classical TLS implementations during the transition period. It's relevant for organizations deploying PQC infrastructure.

Why PQC Downgrade Attacks Matter

Many organizations are deploying hybrid key exchange in TLS 1.3:

  • Kyber (post-quantum) + X25519 (classical)
  • Both are negotiated in the same TLS handshake
  • If either fails, the connection falls back

An attacker can exploit this by:

  1. Intercepting the TLS ClientHello at the network layer
  2. Removing Kyber from the supported curves list
  3. Allowing X25519 to proceed (which is classically breakable)
  4. Recording the encrypted session for future decryption

This is a harvest now, decrypt later attack: the attacker doesn't break the encryption immediately, but stores it until quantum computers become available.

Regulatory Impact: NIST, CISA, and major financial regulators now require organizations to inventory hybrid PQC/classical systems and ensure downgrade protections are in place.

Attack Mechanism

The attack operates at the TLS protocol layer, not the cryptographic layer:

Step 1: Interception

Using tools like iptables or a transparent proxy, capture outbound TLS ClientHello packets.

Step 2: Modification

Parse the ClientHello message structure:

ClientHello {
  protocol_version = TLS 1.3,
  supported_groups = [kyber768, x25519, secp256r1],
  key_share {
    group = kyber768,
    key_exchange = <kyber public key>
  },
  key_share {
    group = x25519,
    key_exchange = <x25519 public key>
  }
}

Remove the Kyber key_share entry entirely. The ClientHello now lists Kyber in supported_groups, but has no actual key material for it.

Step 3: Forwarding

Forward the modified ClientHello to the server. The server, seeing no Kyber key material, falls back to X25519.

Step 4: Recording

Capture the full TLS record including the ServerHello, encrypted application data, and Finished messages. Store for later decryption.

Implementation Details

The POC is built in C using:

  • OpenSSL 3.0+: TLS handshake, X25519 key exchange
  • liboqs: Kyber signature verification and key encapsulation parsing
  • libpcap / raw sockets: Network packet capture and injection

Key Components

Packet Sniffer (libpcap): Monitors network traffic, identifies ClientHello messages by port 443 and TLS record type.

TLS Parser (OpenSSL): Decodes ClientHello structure without decryption (it's unencrypted in TLS 1.3 before handshake completion).

Kyber Extractor (liboqs): Identifies and removes Kyber key encapsulation from the key_share extension.

Packet Injector (raw sockets): Recalculates TLS record checksums and injects the modified packet back into the network.

POC Walkthrough

The attack is demonstrated against a test server running OpenSSL 3.0 with hybrid Kyber+X25519 support:

$ openssl s_server -tls1_3 -groups kyber768:x25519 -cert server.crt -key server.key

$ # On attacker machine:
$ ./pqc_downgrade_attack --interface eth0 --target-port 443

[*] Listening on eth0:443
[*] Intercepted ClientHello from 192.168.1.10
[*] Removing Kyber key_share (group_id=0x220)
[*] Forwarding modified ClientHello
[*] Server accepted X25519 fallback
[*] Recording encrypted session: session.bin (2048 bytes)

The server logs show only X25519 was used, despite the client advertising Kyber:

Ciphers: TLS_AES_256_GCM_SHA384
Key Exchange: X25519
Protocol: TLSv1.3
Verification: The server-side negotiation shows only X25519, confirming the downgrade was successful.

Real-World Implications

For Organizations:

  • Any hybrid PQC/classical system is vulnerable to this attack
  • Attack requires network-layer access (MitM position, compromised router, ISP-level intercept)
  • Encrypted data can be stored and decrypted once quantum computers break X25519
  • Timeline: Years to decades, but the threat is real

For Regulators:

  • NIST's PQC migration guidance assumes clients and servers validate PQC usage
  • Current TLS 1.3 implementations do not detect downgrade attacks on hybrid systems
  • Requires new protocol extensions or cryptographic binding

Detection & Monitoring:

  • Observe ClientHello: does it advertise Kyber but only use X25519?
  • Monitor for clients that support Kyber but never negotiate it (suspicious)
  • Implement TLS middlebox protocol compliance monitoring

Mitigation & Future Work

Short-term Mitigations:

  • Require PQC-only mode (no classical fallback) in sensitive environments
  • Enforce Kyber negotiation; reject connections that don't use it
  • Monitor TLS fingerprints for unexpected X25519-only traffic
  • Use VPN/transport-layer protections against network-level attacks

Long-term Solutions:

  • Cryptographic Binding: Tie the Kyber result to the X25519 result, preventing selective removal
  • Protocol Updates: TLS 1.4 should mandate PQC or reject hybrid downgrade scenarios
  • Attestation: Hardware attestation that PQC was negotiated (e.g., in secure enclaves)
Next Steps: Open-source release of this POC is planned. Researchers and organizations can use it to test their hybrid PQC implementations and validate downgrade protections.
